1. Project Overview
Hotel Billing Software is a full-stack web application designed to digitize and automate hotel operations — from guest registration and room booking through stay management, service tracking, billing and payment collection. The system replaces paper-based processes with a real-time, role-gated platform accessible to hotel staff at all levels.

1.1 Core Objectives
•	Automate end-to-end billing from check-in to check-out
•	Provide real-time room availability and occupancy tracking
•	Record and charge additional services (food, laundry, spa)
•	Support multiple payment modes with partial payment tracking
•	Generate GST-compliant invoices and financial reports
•	Enforce role-based access for Admin, Manager, and Receptionist

2. Database Schemas
The database is organized into 7 MongoDB collections. All schemas use Mongoose ODM. ObjectId references are used for cross-collection relationships. Timestamps (createdAt, updatedAt) are enabled globally on all schemas.

2.1 User Schema
	Collection: users  |  Indexes: email (unique), role
Purpose: Stores hotel staff accounts with role-based access control.

Field	Type	Required	Notes
_id	ObjectId	Auto	MongoDB default primary key
name	String	Yes	Full name of the staff member
email	String	Yes	Unique. Used as login identifier
password	String	Yes	Bcrypt hashed. Min 8 chars
role	Enum	Yes	admin | receptionist | manager
isActive	Boolean	Yes	Soft-delete flag, default true
lastLogin	Date	No	Timestamp of last successful login
createdAt	Date	Auto	Mongoose timestamps
updatedAt	Date	Auto	Mongoose timestamps

2.2 Room Schema
	Collection: rooms  |  Indexes: room_number (unique), availability, category
Purpose: Stores physical room data, tariffs, and current occupancy status.

Field	Type	Required	Notes
_id	ObjectId	Auto	MongoDB default primary key
room_number	String	Yes	Unique room identifier (e.g. '101', 'S-01')
name	String	Yes	Display name (e.g. 'Ocean View Suite')
room_category	Enum	Yes	standard | deluxe | suite
floor	Number	No	Floor number
availability	Enum	Yes	available | occupied | maintenance
price.per_night	Number	Yes	Base nightly rate in INR
price.per_hour	Number	No	Hourly rate (for short stays)
price.tax_percent	Number	Yes	GST percentage (e.g. 12 or 18)
amenities	[String]	No	e.g. ['AC', 'WiFi', 'TV']
max_occupants	Number	Yes	Maximum allowed guests
images	[String]	No	Array of image URLs
bookings	[ObjectId]	Ref	References → bookings collection

2.3 Guest Schema

Purpose: Stores guest personal information, ID proof details, and occupant breakdown.

email	String	No	Optional. Used for invoice emails

documents.number	String	Yes	ID proof number
documents.file_url	String	Yes	S3 / Cloudinary URL of uploaded scan
occupants.total	Number	Yes	Total number of people in room
occupants.adults.count	Number	Yes	Total adults
occupants.adults.male	Number	Yes	Male adults count
occupants.adults.female	Number	Yes	Female adults count

2.4 Booking Schema
	Collection: bookings  |  Indexes: status, check_in_date, room_id
Purpose: Central document linking a room, guest, bill, services and payment for a stay.

Field	Type	Required	Notes
_id	ObjectId	Auto	MongoDB default primary key
room_id	ObjectId	Yes	Reference → rooms._id
guest_id	ObjectId	Yes	Reference → guests._id
bill_id	ObjectId	Yes	Reference → bills._id
check_in_date	Date	Yes	Actual or expected check-in datetime
check_out_date	Date	No	Set at check-out
expected_checkout	Date	Yes	Planned departure datetime
status	Enum	Yes	booked | checked-in | checked-out | cancelled
booking_source	Enum	No	walk-in | online | OTA | phone
advance_paid	Number	No	Advance amount collected at booking
services	[ObjectId]	Ref	Array of references → services._id
notes	String	No	Internal staff notes
created_by	ObjectId	Ref	Reference → users._id (receptionist)

2.5 Service Schema
	Collection: services  |  Indexes: booking_id, type, created_at
Purpose: Individual add-on service charges attached to a booking during a stay.

Field	Type	Required	Notes
_id	ObjectId	Auto	MongoDB default primary key
booking_id	ObjectId	Yes	Reference → bookings._id
type	Enum	Yes	fooding | laundry | spa | other
name	String	Yes	Service item name (e.g. 'Biriyani')
quantity	Number	Yes	Units consumed
unit_price	Number	Yes	Price per unit in INR
total_price	Number	Yes	quantity × unit_price (computed)
description	String	No	Additional details or notes
served_at	Date	Yes	When the service was delivered
added_by	ObjectId	Ref	Reference → users._id (staff)

2.6 Bill Schema
	Collection: bills  |  Indexes: booking_id, status, created_at
Purpose: Aggregated billing document with charges, taxes, discounts, and payment status.

Field	Type	Required	Notes
_id	ObjectId	Auto	MongoDB default primary key
booking_id	ObjectId	Yes	Reference → bookings._id
room_charge	Number	Yes	Calculated from room price × duration
services_charge	Number	Yes	Sum of all services.total_price
tax_amount	Number	Yes	GST computed on (room + service charges)
discount.type	Enum	No	seasonal | loyalty | corporate | manual
discount.amount	Number	No	Discount amount in INR
total_amount	Number	Yes	room + services + tax − discount
amount_paid	Number	Yes	Sum of all successful payments
remaining_amount	Number	Yes	total_amount − amount_paid
status	Enum	Yes	paid | unpaid | partial
payments	[ObjectId]	Ref	Array of references → payments._id
invoice_number	String	Yes	Unique invoice ID (auto-generated)
finalized_at	Date	No	Timestamp when bill was closed

2.7 Payment Schema
	Collection: payments  |  Indexes: bill_id, transaction_id (unique), payment_date
Purpose: Records each individual payment transaction against a bill.

Field	Type	Required	Notes
_id	ObjectId	Auto	MongoDB default primary key
bill_id	ObjectId	Yes	Reference → bills._id
booking_id	ObjectId	Yes	Reference → bookings._id (for fast lookup)
method	Enum	Yes	cash | card | UPI | online_gateway
amount	Number	Yes	Amount paid in this transaction
transaction_id	String	No	Gateway transaction ID (for card/UPI/online)
payment_date	Date	Yes	Timestamp of payment
status	Enum	Yes	success | failed | refunded | pending
refund_amount	Number	No	Amount refunded (if applicable)
refund_date	Date	No	Date of refund
collected_by	ObjectId	Ref	Reference → users._id (staff)
notes	String	No	Optional notes (e.g. 'advance payment')

2.8 Schema Relationship Diagram
	User  ──creates──▶  Booking  ──has one──▶  Guest
Booking  ──belongs to──▶  Room
Booking  ──has many──▶  Services
Booking  ──has one──▶  Bill  ──has many──▶  Payments
Room  ──has many──▶  Bookings  (historical references)
Guest  ──has many──▶  Bookings  (booking history)

5. Business Process Flow

Step 1 — Reservation / Walk-in
•	Guest books room online or walks in to reception
•	System checks room availability for requested date range
•	Booking document is created with status: booked
•	Optional: advance payment is captured, stored in advance_paid field

Step 2 — Check-in
•	Receptionist locates booking or creates new walk-in booking
•	Guest details entered / verified (name, contact, ID proof uploaded)
•	Room assigned — room.availability updated to occupied
•	Booking status updated to checked-in
•	Empty Bill document created and linked to booking

Step 3 — During Stay
•	Staff add service charges as consumed (food, laundry, spa)
•	Each service creates a Service document linked to the booking
•	Bill document is recalculated in real-time on every service addition
•	Receptionist can view running bill at any time from the booking panel

Step 4 — Billing Calculation
•	room_charge = price_per_night × number_of_nights (or hourly rate if applicable)
•	services_charge = sum of all Service.total_price for this booking
•	tax_amount = (room_charge + services_charge) × tax_percent / 100
•	discount deducted if a discount object is present on the bill
•	total_amount = room_charge + services_charge + tax_amount − discount.amount

Step 5 — Check-out & Payment
•	Final bill displayed to guest; receptionist confirms
•	Payment collected via cash / card / UPI / online gateway
•	Each payment creates a Payment document; bill.amount_paid is updated
•	If amount_paid >= total_amount → bill.status = paid
•	If amount_paid > 0 but < total_amount → bill.status = partial
•	Invoice PDF generated and emailed to guest (if email on file)
•	Room.availability updated back to available
•	Booking.status updated to checked-out

Step 6 — Reporting & Audit
•	Daily transaction summary generated via aggregation pipeline
•	Revenue, occupancy, and GST data available on Manager dashboard
•	All critical operations logged to audit_logs collection

7. Security Checklist & Environment Variables

7.1 Security Checklist
•	Passwords hashed with bcrypt (salt rounds ≥ 12)
•	JWT signed with RS256 asymmetric key (not HS256 with shared secret)
•	All inputs validated with Zod before reaching controller
•	File uploads: validate mime type and size before Cloudinary/S3
•	HTTPS enforced in production; HSTS header via helmet
•	MongoDB injection prevented by Mongoose schema typing
•	Rate limiting on /auth/login (max 5 attempts per 15 min per IP)
•	CORS whitelist limited to frontend domain
•	Sensitive fields (password) excluded from all API responses

8. Testing Strategy

8.1 Unit Tests (Jest)
•	Bill calculation utility: test all combinations of room/service/tax/discount
•	Booking status machine: valid and invalid transitions
•	Zod schema validators: valid inputs, missing fields, wrong types
•	Date utility functions: nights calculation, late checkout surcharge

8.2 Integration Tests (Supertest)
•	Full booking lifecycle: create → check-in → add services → checkout → pay
•	Auth flow: login, token expiry, role guard rejection
•	Payment flow: partial payment, second payment, overpayment guard
•	Room availability: concurrent booking conflict detection

8.3 End-to-End Tests (Playwright)
•	Receptionist walk-in flow in browser
•	Bill generation and PDF download
•	Manager applies discount and views revenue report

Appendix — GST Rate Reference
Room Tariff (per night)	GST Rate	Note
< ₹1,000	0%	No GST applicable
₹1,000 – ₹7,499	12%	Standard hotel room GST slab
≥ ₹7,500	18%	Luxury and suite category
Restaurant (hotel)	5%	Food & beverages (no ITC)
Spa / Wellness	18%	Spa and salon services

	Note: GST rates are subject to change. Always verify the current slabs from the GST Council notification.
Invoice must include: GSTIN of hotel, HSN/SAC code, tax breakup (CGST + SGST or IGST), and sequential invoice number.
